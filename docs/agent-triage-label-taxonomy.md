# Agent-first triage label taxonomy

This document defines the minimum organization-level metadata contract for SylphxAI agent-first GitHub triage.

It is intentionally small. Repositories may keep their own product, component, area, and release labels; Signal/Triage agents must not overwrite those repo-owned semantics. The goal is to make cross-repo stale/blocked/CI/risk queues mechanically discoverable while preserving project boundaries.

## Scope

Applies to active SylphxAI repositories that opt into agent-first workflow triage.

Signal/Triage agents may use this taxonomy to propose or apply labels when the classification is evidence-backed by GitHub issue/PR state, CI/check state, repository docs, or explicit manager/project instruction.

Signal/Triage agents must still read the repository boundary first where present, in this order:

1. `PROJECT_BOUNDARY.md`
2. `AGENT_GUIDE.md`
3. `AGENTS.md`
4. `README.md`
5. relevant ADR/spec/docs called out by the issue or PR

## Guardrails

Agent-first triage labels are metadata, not authority to merge, deploy, or bypass review.

Agents must not:

- mix work across repositories or projects;
- force-push or push directly to protected/default branches;
- change destructive repository settings;
- expose secrets or credential material;
- merge or deploy without the project-required gates and manager-visible evidence;
- create broad label churn when the repository already has a more specific project-owned convention.

## Minimum labels

### State labels

Use state labels for current routing state. Remove or replace stale state labels when they stop being true.

| Label | Meaning | Evidence examples |
| --- | --- | --- |
| `state/blocked` | Cannot progress without another PR, decision, failing dependency, merge conflict, or external owner. | Dirty merge state, explicit blocker comment, required upstream issue/PR. |
| `state/stale` | No meaningful activity for the repository's stale threshold. | `updatedAt` older than threshold and no active owner signal. |
| `state/ready` | Ready for the next human/agent action. | Checks green, scope clear, no known blocker. |
| `state/needs-scope` | Needs clearer acceptance criteria, owner, boundary, or project metadata. | Vague issue, missing repro, missing target repo/service. |
| `state/ci-failing` | CI/checks are failing and the failure appears actionable for this PR/issue. | Failed required GitHub checks or named workflow failures. |

### Role/source labels

Use role/source labels to identify who or what originated the signal.

| Label | Meaning |
| --- | --- |
| `role/signal-triage` | Signal/Triage agent touched or owns the metadata/routing signal. |
| `source/agent` | Created or materially updated by an agent. |
| `source/human` | Created or materially updated by a human. |
| `source/dependabot` | Generated by Dependabot or dependency automation. |

### Type labels

Use type labels for work category. Repositories may map these to existing labels (`bug`, `enhancement`, `documentation`, etc.) during rollout.

| Label | Meaning |
| --- | --- |
| `type/bug` | Defect or incorrect behavior. |
| `type/enhancement` | New capability or improvement. |
| `type/docs` | Documentation-only work. |
| `type/chore` | Maintenance, configuration, or housekeeping. |
| `type/refactor` | Internal structure change without intended behavior change. |
| `type/security` | Security-sensitive fix or hardening. |
| `type/performance` | Performance, latency, throughput, capacity, or cost work. |

### Priority and risk labels

Use priority/risk labels only when evidence is clear or the owning project/manager has stated priority.

| Label | Meaning |
| --- | --- |
| `priority/P0` | Immediate incident, production outage, active security issue, or hard release blocker. |
| `priority/P1` | High-priority blocker or user-visible defect. |
| `priority/P2` | Normal priority planned work. |
| `priority/P3` | Low-priority backlog or cleanup. |
| `risk/security` | Could affect authentication, authorization, secrets, supply chain, or data exposure. |
| `risk/prod` | Could affect production runtime, deploy, CI/release, or live customer paths. |
| `risk/data` | Could affect data integrity, migration, retention, or privacy. |
| `risk/customer` | Customer-facing behavior, billing, support, onboarding, or API contract risk. |

### Area labels stay repo-owned

Area labels must remain close to the repository's architecture and docs.

Examples:

- `area/platform`
- `area/worker`
- `area/ci`
- `area/infra`
- `area/storage`
- `area/auth`
- `area/docs`

Do not apply an area label unless the repository already defines it or the owning project has accepted it.

## Triage comment convention

Prefer labels over comments for routine classification.

Add at most one Signal/Triage comment when it adds coordination value, such as:

- summarizing a queue-wide blocker;
- identifying duplicates with links;
- recording a stale/blocked evidence bundle;
- proposing a safe next owner/action;
- clarifying why a PR should be handled before dependent queue items.

A Signal/Triage comment should include:

1. what was observed;
2. evidence links or check names;
3. the recommended next routing action;
4. an explicit note when no merge/deploy/code action was taken.

## Rollout model

1. Adopt this document as the org SSOT for minimum triage metadata.
2. Roll out labels repo-by-repo, preserving existing project labels.
3. When a repository has equivalent existing labels, map to those labels instead of duplicating.
4. Add automation only after the label mapping is stable.
5. Treat stale/blocked reports as derived from GitHub state plus these labels, not free-form memory.

## Pilot evidence

The first write-capable Agent-first pilot scan on 2026-06-18 found:

- `platform`: 50 open PRs and many missing labels despite a rich repo taxonomy.
- `spiron`: 11 open PRs with only default labels available.
- `filesystem-mcp`: 20 open PRs blocked by the same `Validate Code Quality` check family.
- `tryit`: 188 open issues with rich but inconsistent historical labels.
- `bgca`: 5 open PRs with only default labels available.

This taxonomy is the minimum process fix needed before broad cross-repo label automation.
